# RELIABILITY.md
# KakaoCloud Essential Basic Course — 정답 검증 프로세스

> 이 문서는 문제은행의 신뢰성을 유지하기 위한 정답 검증 및 오류 처리 프로세스를 정의합니다.
> "이 문제의 정답이 틀린 것 같다"는 상황이 발생했을 때 이 문서를 따릅니다.

---

## 1. 신뢰성 원칙

문제은행의 신뢰성은 두 가지 축으로 구성됩니다.

**정확성** — 정답이 공식 문서 기준으로 올바른가
**최신성** — 공식 문서가 변경됐을 때 문제가 적시에 갱신되는가

모든 active 문제는 `source` 필드의 공식 문서 URL로 정답을 역추적할 수 있어야 합니다.
역추적이 불가능한 문제는 신뢰성이 없는 문제입니다.

---

## 2. 오류 신고 프로세스

스터디 중 또는 리뷰 중 정답 오류가 의심될 때 아래 절차를 따릅니다.

```
[1] 의심 발견
    팀원이 스터디 중 "이 정답이 틀린 것 같다"고 판단

    ↓

[2] 공식 문서 1차 확인
    문제의 source URL에 직접 접근하여 정답 근거 확인
    → 공식 문서가 변경되어 정답이 달라진 경우:  [4]번으로
    → 문제 자체의 오류인 경우:                   [3]번으로

    ↓

[3] GitHub Issue 또는 PR 생성
    제목:  [오류신고] q{NNN} — {오류 내용 한 줄 요약}
    내용:
      - 문제 ID: q{NNN}
      - 현재 정답: {A/B/C/D}
      - 의심 근거: {공식 문서 URL 또는 설명}
      - 수정 제안: {수정 후 정답 또는 해설}

    ↓

[4] 팀 논의 (최소 2인 확인)
    신고자 외 1인 이상이 공식 문서를 직접 확인하여 동의
    → 동의: [5]번으로
    → 이견: 추가 논의 후 재결정

    ↓

[5] 수동 수정 (사람만 가능)
    해당 q{NNN}.md 파일 수정:
      - 정답 변경
      - 해설 업데이트
      - source URL 갱신
      - reviewed_by 업데이트
      - 변경 이유를 PR 설명에 기록

    ↓

[6] validate-quality.py 재실행
    수정된 문제 재채점 → quality_score 갱신
```

> **주의**: 정답 변경은 반드시 PR로만 가능합니다.
> 직접 main 브랜치에 push하여 정답을 수정하는 행위는 금지됩니다.

## 3. 공식 문서 변경 감지 시 처리

`parse-kakao-docs.py` 실행 후 변경이 감지된 경우:

```
[1] parse-kakao-docs.py 가 변경 감지 경고 출력
    [WARN] docs/references/bcs-llms.txt 변경 감지
           영향 가능 문제: q004, q011, q023

    ↓

[2] 팀 알림 (Slack/이슈)
    영향 가능 문제 목록을 팀에 공유

    ↓

[3] 팀원이 해당 문제들을 수동 확인
    각 문제의 source URL 접근 → 정답 여전히 유효한지 검토

    ↓

[4a] 정답 여전히 유효 → 유지 (변경 없음)
[4b] 정답 변경 필요 → [오류 신고 프로세스] §2의 [5]번부터 진행
[4c] 문제 자체 폐기 판단 → status를 deprecated로 수동 변경
```

## 4. 정기 검증 스케줄

| 주기 | 작업 | 담당 |
|------|------|------|
| 월 1회 | `parse-kakao-docs.py --all` 실행 → 변경 감지 확인 | 스크립트 담당 |
| 분기 1회 | active 문제 전체 `source` URL 유효성 검사 | 팀 전체 |
| 스터디 후 | 오답률 60% 이상 문제 → `tech-debt-tracker.md` 기록 | 스터디 진행자 |

## 5. 오답률 기반 문제 품질 재검토

스터디 결과에서 특정 문제의 오답률이 높을 경우, 두 가지 가능성을 검토합니다.

**가능성 A** — 문제가 어렵지만 올바름 (학습 효과 있는 문제)
→ `tech-debt-tracker.md` 에 기록하고 해설 강화 검토

**가능성 B** — 문제 자체에 모호성 또는 오류가 있음
→ 오류 신고 프로세스 진행 (§2)

**판단 기준**

| 오답률 | 조치 |
|--------|------|
| 40~60% | 해설 보강 검토 |
| 60% 이상 | 문제 재검토 필수. 모호성·오류 여부 팀 논의 |
| 80% 이상 | 문제 일시 비활성화(`review` 강등) 후 재작성 검토 |

## 6. deprecated 문제 관리

`status: deprecated` 로 변경된 문제는 아래와 같이 관리합니다.

- `build-exam.py` 가 deprecated 문제를 자동으로 제외
- 파일 자체는 삭제하지 않고 보존 (히스토리 추적 목적)
- deprecated 사유를 frontmatter에 기록

```yaml
status: deprecated
deprecated_reason: "2026-05 공식 문서 변경으로 정답 기준 소실"
deprecated_at: 2026-05-01
```
