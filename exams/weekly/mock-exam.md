## 정답 및 해설

### 1. C
**해설**: 카카오클라우드의 데이터베이스는 스냅샷 백업과 바이너리 로그를 함께 활용하여 특정 시점으로 데이터베이스를 복구할 수 있습니다.
스냅샷은 복구의 기준점 역할을 하고, 바이너리 로그는 스냅샷 이후 발생한 변경 사항을 추적하여 원하는 정확한 시점까지 데이터를 재현하는 데 사용됩니다.
**오답 포인트**:
- A: VPC 피어링은 네트워크 연결 기능으로 데이터베이스 복구 메커니즘과 무관합니다.
- B: Read Replica는 읽기 부하 분산을 위한 복제본이며, PITR의 핵심 구성 요소가 아닙니다.
- D: Object Storage 버킷과 보안 그룹은 스토리지 및 접근 제어 서비스로, 데이터베이스 시점 복구의 핵심 기술이 아닙니다.

<br>

### 2. B
**해설**: 카카오클라우드 Kubernetes Engine은 VPC 기반 위에서 쿠버네티스를 손쉽게 실행할 수 있도록 지원하는 관리형(Managed) 쿠버네티스 서비스입니다.
관리형 서비스이므로 마스터 노드의 운영·관리 부담 없이 클러스터를 생성하고 애플리케이션을 배포할 수 있습니다.
**오답 포인트**:
- A: Kubernetes Engine은 베어메탈 전용이 아닌 VPC 기반의 가상화 환경에서 동작합니다.
- C: 단일 가용 영역 제한은 Kubernetes Engine의 특징이 아닙니다.
- D: 관리형 서비스이므로 사용자가 마스터 노드를 직접 설치하거나 운영할 필요가 없습니다.

<br>

### 3. B
**해설**: SmartNIC은 네트워크 패킷 처리, 스토리지 I/O 등의 작업을 호스트 CPU 대신 별도의 전용 칩이 처리하도록 오프로드하는 기술입니다.
이를 통해 호스트 CPU의 부담이 줄어들어 인스턴스에서 실행되는 애플리케이션이 더 많은 컴퓨팅 자원을 활용할 수 있게 됩니다.
**오답 포인트**:
- A: CPU 코어 수 동적 확장은 SmartNIC의 역할이 아니라 Scale-up 또는 CPU 오버프로비저닝 관련 기능입니다.
- C: GPU 연산 오프로드는 SmartNIC이 아닌 GPU 가속기(CUDA 등)의 역할입니다.
- D: 전력 절감을 위한 패킷 압축은 SmartNIC의 주된 기능 정의와 다르며, 핵심 목적은 컴퓨팅 성능 향상입니다.

<br>

### 4. B
**해설**: 카카오클라우드 공식 문서에 따르면, Monitoring의 Metric Export 엔드포인트를 이용하여 서드파티 모니터링 도구에서 카카오클라우드의 모니터링 데이터를 확인할 수 있습니다.
Metric Export는 외부 시스템이 카카오클라우드 메트릭 데이터를 수집·활용할 수 있도록 표준화된 방식으로 데이터를 제공하는 기능입니다.
**오답 포인트**:
- A: Metric Import는 외부에서 데이터를 가져오는 방향으로, Monitoring 데이터를 서드파티로 내보내는 기능이 아닙니다.
- C: Log Stream 파이프라인은 로그 데이터 전송에 관련된 개념으로, 메트릭 데이터 내보내기와는 다릅니다.
- D: Alert Webhook 채널은 알림 발송에 사용되는 기능으로, 모니터링 데이터를 외부로 내보내는 기능이 아닙니다.

<br>

### 5. D
**해설**: 카카오클라우드 데이터베이스는 자동 백업(스냅샷)과 바이너리 로그를 함께 활용하여 원하는 시점으로 데이터베이스를 복구할 수 있습니다.
스냅샷은 특정 시점의 전체 데이터를 보존하고, 바이너리 로그는 그 이후 발생한 변경 사항을 추적하여 세밀한 시점 복구(PITR)를 가능하게 합니다.

### 6. A

### 8. C

사용자가 필요한 만큼 가상 서버를 생성하고 사용할 수 있도록 지원하는 것이 핵심 역할이며, 이는 컴퓨팅 자원 제공을 목적으로 합니다.
**오답 포인트**:
- A: 네트워킹 서비스 그룹은 VPC, 로드밸런서 등 네트워크 관련 서비스를 포함하며 BCS는 이에 해당하지 않습니다.
- B: 스토리지 서비스 그룹은 Object Storage, Block Storage 등을 포함하며 BCS의 분류가 아닙니다.
- D: 데이터베이스 서비스 그룹은 RDS 등 DB 관련 서비스를 포함하며 BCS와 무관합니다.

### 9. B

이는 고가용성(HA, High Availability)과 재해 복구(DR) 전략의 핵심 구성 요소로, 서비스 연속성을 보장하는 데 목적이 있습니다.
**오답 포인트**:
- A: 멀티 AZ는 가용성 향상을 위한 것이며, 개별 인스턴스의 CPU 성능을 향상시키는 기술이 아닙니다.
- C: 퍼블릭 IP와 멀티 AZ는 별개의 개념으로, AZ 구성이 외부 인터넷 통신 방식에 영향을 주지 않습니다.
- D: 멀티 AZ는 여러 AZ에 자원을 분산 배치하므로 오히려 비용이 증가할 수 있으며, 비용 절감을 목적으로 하지 않습니다.

### 10. B

이를 통해 데이터 증가에 따른 스토리지 확장을 인스턴스 재생성 없이 유연하게 대응할 수 있습니다.
**오답 포인트**:

- C: 블록 스토리지는 동일 리전 및 가용 영역에 위치해야 인스턴스에 연결할 수 있습니다.
- D: 일부 블록 스토리지 유형은 멀티-어태치(Multi-Attach)를 지원하여 여러 인스턴스에서 접근할 수 있는 경우도 있습니다.

- D: 서버리스 아키텍처는 특정 이벤트 기반 워크로드에 적합한 구성으로, 카카오클라우드 입문용 기본 시나리오가 아닙니다.

- C: 경량 하이퍼바이저를 사용한다는 것은 가상화가 존재한다는 의미이므로, Bare Metal의 정의에 위배됩니다.
- D: BCS Bare Metal도 클라우드 콘솔을 통해 온디맨드 프로비저닝이 가능한 것이 클라우드 서비스의 핵심 특징입니다.

### 14. A

**오답 포인트**:

- C: BCS는 카카오클라우드가 인프라를 관리하며, 사용자가 직접 데이터센터를 운영하지 않습니다.
- D: BCS는 인터넷 기반 클라우드 서비스이며, 내부 네트워크 전용 서버 서비스가 아닙니다.

### 16. B

- C: 버스터블 인스턴스는 GPU 가속과 관련이 없으며, 범용 컴퓨팅 워크로드를 대상으로 합니다.
- D: 버스터블 인스턴스는 서버리스 환경이 아닌 일반 VM 기반 인스턴스 유형입니다.

### 18. B

HTTP 기반 RESTful API 방식으로 접근하므로, 별도의 파일시스템 마운트나 블록 디바이스 연결 없이도 다양한 애플리케이션에서 유연하게 활용할 수 있습니다.

- A: Object Storage는 NFS 전용이 아니며, API를 통한 HTTP 방식으로 접근합니다.
- C: Object Storage는 블록 디바이스 방식이 아닌 오브젝트 방식으로 데이터를 저장합니다.
- D: Object Storage는 내구성과 가용성을 위해 다중 복제 저장 방식을 채택하고 있습니다.

이를 통해 애플리케이션에서 자동화된 방식으로 파일 업로드, 다운로드, 삭제 등 다양한 스토리지 작업을 수행할 수 있습니다.
**오답 포인트**:
- A: Object Storage는 블록 디바이스 방식이 아닌 객체(Object) 단위로 데이터를 관리합니다.
- B: NFS 전용이 아니라 HTTP 기반 REST API를 통해 접근하는 방식을 사용합니다.
- D: Object Storage는 특정 인스턴스에 마운트하는 로컬 스토리지가 아닌 범용 분산 스토리지입니다.
