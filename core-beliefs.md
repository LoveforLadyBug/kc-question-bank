## 1. 좋은 문제의 5가지 조건

### 1-1. 공식 문서에 근거한다

문제의 정답과 해설은 반드시 `docs.kakaocloud.com` 에서 확인 가능한 내용이어야 합니다.
시험 당일 기준으로 정답이 바뀔 수 있는 추측성 문제, 버전에 따라 달라지는 내용,
공식 문서에 명시되지 않은 내부 구현 세부사항은 출제하지 않습니다.

```
✅ "카카오클라우드 Object Storage가 지원하는 API 호환 방식은?"
   → 공식 문서에 S3 호환 API 명시
❌ "카카오클라우드 내부 스토리지 노드 복제 방식은?"
   → 공식 문서에 없는 내부 구현
```

### 1-2. 한 문제는 하나의 개념만 테스트한다

한 문제 안에 두 개 이상의 판단이 필요한 복합 조건을 넣지 않습니다.
복잡한 시나리오가 필요하다면 문제를 분리합니다.

```
✅ "VPC의 역할로 올바른 것은?"
❌ "VPC를 생성하고 Subnet을 나눈 뒤 Load Balancer를 연결할 때
    올바른 순서와 각 서비스의 역할을 고르시오."
    → 두 개의 문제로 분리해야 함
```

### 1-3. 오답은 "틀릴 만한 이유"가 있어야 한다

오답 보기는 단순한 반대말이나 무관한 내용이 아니라,
실제로 헷갈릴 수 있는 관련 개념이어야 합니다.
좋은 오답은 "왜 틀렸는지"를 설명할 수 있는 오답입니다.

```
✅ 오답으로서의 좋은 보기 예시 (BNS 문제에서)
   - "Load Balancer" ← 네트워크 관련이지만 역할이 다름
   - "Transit Gateway" ← 같은 카테고리지만 용도가 다름

❌ 오답으로서의 나쁜 보기 예시
   - "데이터베이스" ← 네트워크 문제에 전혀 무관한 보기
   - "아무것도 아님" ← 무의미한 보기
```

### 1-4. 해설은 정답뿐 아니라 오답의 이유를 설명한다

정답만 맞히는 것이 목표가 아니라, 오답을 선택했을 때도 배움이 있어야 합니다.
모든 오답 보기에 대해 "왜 이것이 틀렸는지"를 `## 오답 포인트` 섹션에 작성합니다.

```
✅ 좋은 해설
   ## 오답 포인트
   - B. Load Balancer: 트래픽 분산 역할이며, 네트워크 격리와는 무관합니다.
   - C. Transit Gateway: VPC 간 연결에 사용하며, 격리된 네트워크 공간 생성이 목적이 아닙니다.

❌ 나쁜 해설
   ## 오답 포인트
   - B. 틀림
   - C. 틀림
```

### 1-5. 스터디에서 "왜?"를 물을 수 있는 문제를 만든다

단순 암기형 문제보다, 팀원이 "왜 A가 정답이고 B는 아닌가?"를 토론할 수 있는
문제가 좋은 스터디 문제입니다. 해설을 읽고 끝나는 문제가 아니라,
해설을 읽고 나서 더 깊은 질문이 생기는 문제를 지향합니다.

## 2. 하지 말아야 할 것들

| 금지 유형 | 이유 |
|-----------|------|
| "다음 중 틀린 것은?" 형식 | 부정 문항은 혼란을 유발하고 학습 효과가 낮음 |
| 보기에 "모두 맞다 / 모두 틀리다" 포함 | 추론이 아닌 요행에 의존하게 됨 |
| 정답 보기만 유독 길게 작성 | 보기 길이가 힌트가 되어 문제 의미 훼손 |
| 공식 문서 문장을 그대로 복사한 문제 | 이해가 아닌 문장 암기를 유도 |
| 서비스 이름만 바꾼 반복 문제 | 문제은행 다양성 저하 |
| 정답이 카카오 서비스로 유도되는 광고성 문제 | 학습 객관성 훼손 |

## 3. 문제 유형 정의

이 프로젝트는 **4개 보기 단일선택(single)** 유형만 사용합니다.

### 선택 이유

팀 스터디 맥락에서 단일선택은 명확한 정답 기준을 제공하고,
스터디 후 결과 비교와 오답 분석이 용이합니다.
OX나 다중선택은 향후 필요 시 이 문서를 개정하여 추가할 수 있습니다.

### 유형 명세

```yaml
type: single

보기 수: 4개 (A / B / C / D)
정답 수: 1개
보기 길이: 각 보기는 한 줄 이내 권장 (최대 2줄)
보기 순서: 논리적 순서 또는 길이 순 정렬 (정답 위치 편향 방지)
```

### 보기 순서 편향 방지 규칙

정답이 항상 A나 B에 몰리지 않도록, `generate-questions.py` 실행 시
정답 위치를 A~D 사이에서 균등하게 배분합니다.
챕터 내 정답 분포는 `_index.md` 에서 추적합니다.

## 4. 난이도 정의

난이도는 **1~5단계**를 사용합니다.

| 단계 | 레이블 | 정의 |
|------|--------|------|
| 1 | 기초 I | 서비스/개념의 이름과 정의를 아는가 |
| 2 | 기초 II | 서비스의 주요 특징과 타 서비스와의 차이를 아는가 |
| 3 | 응용 | 서비스를 언제, 왜 쓰는지 판단할 수 있는가 |
| 4 | 심화 I | 여러 서비스를 조합한 아키텍처 결정을 할 수 있는가 |
| 5 | 심화 II | 제약 조건이나 비용을 고려한 최적 선택을 할 수 있는가 |

### 챕터별 난이도 분포 권장

| 난이도 | 권장 비율 | 비고 |
|--------|-----------|------|
| 1~2 | 40% | 기초 개념 확인 |
| 3 | 35% | 핵심 응용 |
| 4~5 | 25% | 심화 및 모의시험 활용 |

> 모의시험 세트 (`build-exam.py --mode mock`) 는 난이도 3~5 문제를 우선 포함합니다.
> 주차별 스터디 세트는 해당 챕터의 분포 권장 비율을 따릅니다.
