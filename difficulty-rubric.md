# KakaoCloud Essential Basic Course — 난이도 판단 기준

> 이 문서는 문제를 만들거나 리뷰할 때 난이도(1~5)를 일관되게 판단하기 위한
> 실전 기준표입니다.
>
> "이 문제는 몇 난이도인가?" 판단이 애매할 때 이 문서를 펼치세요.

---

## 1. 난이도 한눈에 보기

| 난이도 | 레이블 | 핵심 질문 | 응시자가 할 수 있어야 하는 것 |
|--------|--------|-----------|---------------------------|
| 1 | 기초 I | "이게 뭐야?" | 서비스/용어의 이름과 정의를 안다 |
| 2 | 기초 II | "이게 어떻게 달라?" | 유사 서비스와의 차이를 구분한다 |
| 3 | 응용 | "이걸 언제 써?" | 상황에 맞는 서비스를 선택할 수 있다 |
| 4 | 심화 I | "이걸 어떻게 조합해?" | 여러 서비스를 연결한 구조를 설계한다 |
| 5 | 심화 II | "이게 최선이야?" | 제약과 비용을 고려해 최적 선택을 한다 |

---

## 2. 난이도별 상세 기준 및 예시

---

### 난이도 1 — 기초 I "정의를 안다"

**판단 기준**
- 서비스 이름, 약어, 정의를 직접 묻는다
- 공식 문서의 첫 번째 단락 수준의 지식이면 답할 수 있다
- 보기가 서로 완전히 다른 카테고리의 개념이다

**예시 문제**
```
카카오클라우드에서 논리적으로 격리된 가상 네트워크 공간을 제공하는 서비스는?

A. VPC
B. Load Balancer
C. Object Storage
D. IAM
```
→ VPC의 정의만 알면 풀 수 있다. 보기가 모두 다른 카테고리.

---

### 난이도 2 — 기초 II "차이를 안다"

**판단 기준**
- 같은 카테고리 내 서비스들을 구분해야 한다
- "A와 B의 차이는?", "A의 특징으로 올바른 것은?" 형식
- 보기가 같은 카테고리 안의 유사 서비스들로 구성된다

**예시 문제**
```
카카오클라우드 스토리지 서비스 중, S3 호환 API를 지원하며
대규모 비정형 데이터를 객체 형태로 저장하는 서비스는?

A. Block Storage
B. Object Storage
C. File Storage
D. Archive Storage
```
→ 세 스토리지 타입의 특징 차이를 알아야 한다.

---

### 난이도 3 — 응용 "언제 쓰는지 안다"

**판단 기준**
- 특정 요구사항이나 상황이 주어지고, 적합한 서비스를 선택해야 한다
- "~한 상황에서 가장 적합한 서비스는?" 형식
- 정답을 맞히려면 서비스의 정의뿐 아니라 용도와 제약을 알아야 한다

**예시 문제**
```
여러 Virtual Machine 인스턴스에서 동시에 읽기/쓰기가 필요한
공유 스토리지를 구성하려 한다. 가장 적합한 서비스는?

A. Block Storage
B. Object Storage
C. File Storage
D. GPU 인스턴스 로컬 디스크
```
→ "공유 가능 여부"라는 요구사항을 각 스토리지 특성에 매핑해야 한다.

---

### 난이도 4 — 심화 I "서비스를 조합할 수 있다"

**판단 기준**
- 두 개 이상의 서비스가 등장한다
- 아키텍처 구성 또는 서비스 연결 순서/방식을 묻는다
- 각 서비스를 개별로 알아도, 조합 맥락을 모르면 틀릴 수 있다

**예시 문제**
```
외부 인터넷에서 접근 가능한 웹 서버를 카카오클라우드에 구성하려 한다.
반드시 필요한 구성 요소의 조합으로 올바른 것은?

A. VPC + Subnet + Virtual Machine + Public IP
B. VPC + Virtual Machine + Object Storage
C. Subnet + Load Balancer + Container Registry
D. Transit Gateway + Virtual Machine + File Storage
```
→ 웹 서버 외부 노출을 위한 필수 구성 요소들의 역할을 조합해서 판단해야 한다.

---

### 난이도 5 — 심화 II "최적 선택을 할 수 있다"

**판단 기준**
- 여러 선택지가 "모두 가능하지만 하나가 더 낫다"는 형태
- 비용, 성능, 가용성, 제약 조건 중 하나 이상이 판단 기준이 된다
- 실무 경험이 있어야 직관적으로 풀 수 있는 수준

**예시 문제**
```
트래픽이 급증하는 이벤트 기간에만 대규모 컴퓨팅 자원이 필요하고,
평상시에는 소규모 인스턴스로 운영하는 서비스를 설계한다.
비용 효율성을 최우선으로 고려할 때 가장 적합한 전략은?

A. 최대 트래픽 기준의 대형 인스턴스를 항상 유지한다
B. 종량제 과금의 VM을 이벤트 기간에만 추가 생성하고 종료한다
C. Bare Metal Server를 약정 계약으로 확보해 비용을 절감한다
D. GPU 인스턴스를 사용해 컴퓨팅 효율을 높인다
```
→ 종량제 과금 모델의 특성과 이벤트성 트래픽 패턴을 연결해야 한다.

## 3. 애매한 경우 판단 규칙

난이도 경계에서 헷갈릴 때 아래 규칙을 적용합니다.

### 1 vs 2 구분

```
보기가 서로 다른 카테고리 → 난이도 1
보기가 같은 카테고리 내 유사 서비스 → 난이도 2
```

### 2 vs 3 구분

```
"이 서비스의 특징은?" → 난이도 2
"이 상황에서 어떤 서비스를 써야 하는가?" → 난이도 3
```

### 3 vs 4 구분

```
서비스 1개의 적합한 선택 → 난이도 3
서비스 2개 이상의 조합 판단 → 난이도 4
```

### 4 vs 5 구분

```
"어떻게 구성하는가?" (방법) → 난이도 4
"왜 이것이 더 나은가?" (비용/성능/제약 근거) → 난이도 5
```

## 4. 난이도 분포 자동 경고

`validate-quality.py` 는 챕터별 난이도 분포를 검사하여
아래 기준을 벗어나면 `_review-notes.md` 에 경고를 추가합니다.

| 경고 조건 | 메시지 |
|-----------|--------|
| 난이도 1~2 비율이 60% 초과 | `[WARN] 기초 문제 비율 과다 — 응용/심화 문제 추가 권장` |
| 난이도 4~5 비율이 40% 초과 | `[WARN] 심화 문제 비율 과다 — 기초/응용 문제 추가 권장` |
| 난이도 3 문제가 0개 | `[WARN] 응용 문제 없음 — 난이도 3 문제 1개 이상 필요` |
